# Архитектура Founders Vision Alignment Tool

Описание для обсуждения с Claude: где что хранится, хостинг, роли, сценарии. Без кода.

---

## 1. Роли

- **Админ** — тот, кто один раз открыл приложение по ссылке с секретом (`?admin=SECRET`). Может создавать новые комнаты, видеть список всех комнат с данными, переключаться между комнатами, открывать логи. Статус админа хранится в `sessionStorage` браузера до закрытия вкладки.
- **Таня** — один из двух со-основателей. Заполняет свою анкету в комнате, видит только свои ответы до момента сравнения.
- **Алена** — второй со-основатель. Аналогично: свои ответы, до сравнения данные партнёра не видит.

Роли «Таня» и «Алена» задаются через ссылку (`?user=tanya` или `?user=alena`). Если пользователь зашёл по такой ссылке, он не может переключиться на другого пользователя (заблокирован на «своей» роли).

---

## 2. Хостинг и окружение

- **Фронтенд:** статический SPA (React + Vite), деплой на **Vercel**. Пользователи открывают приложение по публичному URL (например `https://stations-rho.vercel.app`).
- **Бэкенд данных:** **Supabase** (приоритет). Таблица в PostgreSQL, доступ по REST API с anon-ключом. Опционально запасной вариант — Firebase Firestore; если нет ни Supabase, ни Firebase — данные только в `localStorage` текущего устройства.
- **Внешние API:** **OpenRouter** для чата с Claude в песочнице (модель Claude Sonnet 4.5). Ключ задаётся в переменных окружения, без ключа песочница недоступна.
- **Переменные окружения:** задаются в `.env.local` локально и в настройках проекта на Vercel для продакшена. Примеры: `VITE_SUPABASE_URL`, `VITE_SUPABASE_ANON_KEY`, `VITE_OPENROUTER_API_KEY`, `VITE_ADMIN_SECRET`. Никакие секреты не хранятся в репозитории.

---

## 3. Хранилища данных

### 3.1 Supabase (основной бэкенд)

**Проект:** создаётся вручную в Supabase Dashboard. Регион на выбор (например Frankfurt).

**Таблица `vision_rooms`:**

| Поле         | Тип       | Описание |
|-------------|-----------|----------|
| `room_id`   | TEXT, PK  | Уникальный идентификатор комнаты (UUID). |
| `tanya_data`| JSONB     | Ответы Тани: объект вида `{ "q1": "...", "q2": [...], ... }`. Ключи — id вопросов из опросника. |
| `alena_data`| JSONB     | Ответы Алены в том же формате. |
| `tanya_chat`| JSONB     | История чата с Claude у Тани в этой комнате. Массив сообщений `{ role, content }`. По умолчанию `[]`. |
| `alena_chat`| JSONB     | Аналогично — чат Алены с Claude. |
| `ai_analysis`| JSONB     | Результат AI-анализа совместимости (скор 0–100, вердикт, сильные стороны, расхождения, вопросы для обсуждения, резюме). Заполняется при первом открытии сравнения. |
| `ai_analysis_at`| TIMESTAMPTZ | Время последнего запуска AI-анализа. Если `updated_at` позже — анализ считается устаревшим. |
| `updated_at`| TIMESTAMPTZ | Время последнего обновления строки (ответы или чаты). |

Доступ к таблице: анонимный read/write через RLS-политики (разрешены select, insert, update для всех без авторизации). Отдельной аутентификации пользователей нет — «кто есть кто» определяется только параметром `user` в URL и тем, в какое поле пишут (tanya_data vs alena_data).

### 3.2 Firebase Firestore (запасной вариант)

Коллекция `vision_rooms`, документ id = `roomId`. Поля: `tanyaData`, `alenaData`, `updatedAt` (в camelCase; данные в виде JSON-строк). Чаты в этой схеме в текущей реализации через Firestore не персистятся.

### 3.3 localStorage (fallback без облака)

Ключи вида `vision_room_{roomId}_tanya`, `vision_room_{roomId}_alena`, `vision_room_{roomId}_tanya_chat`, `vision_room_{roomId}_alena_chat`. Значения — JSON-строки. Работает только на одном устройстве и одном браузере.

## 4. Идентификация и навигация

- **Комната:** задаётся параметром URL `room` (UUID). Без комнаты в ссылке не-админ видит экран «Нет комнаты» и может вставить ID комнаты вручную. Админ при первом заходе без `room` получает новую комнату (генерируется UUID и подставляется в URL).
- **Пользователь в комнате:** параметр `user=tanya` или `user=alena`. Если в ссылке указан `user`, этот пользователь не может сменить роль (экран выбора пользователя показывает только «продолжить как я», смена на другого отключена).
- **Экран:** параметр `screen=comparison` даёт прямой переход на страницу сравнения для данной комнаты и пользователя. Остальные экраны (выбор пользователя, приветствие, заполнение) не кодируются в URL явно, только через состояние приложения.
- Админский вход: `?admin=SECRET`. После проверки секрет убирается из URL, флаг админа пишется в `sessionStorage`.

## 5. Пользовательские сценарии

### 5.1 Админ создаёт комнату и приглашает

1. Открывает сайт с `?admin=SECRET`.
2. Попадает на экран выбора пользователя с текущей комнатой (или создаётся новая, если room не было в URL).
3. Видит блок «Комнаты, где есть данные» — список комнат из Supabase, в которых есть хотя бы какие-то ответы (Таня и/или Алена), со статусами «не начала / заполняет / закончила» и возможностью перейти в любую комнату по клику.
4. Может нажать «Создать новую» — генерируется новый room_id, URL обновляется, страница перезагружается.
5. Копирует «Ссылка для Тани» и «Ссылка для Алены» и отправляет партнёрам.

### 5.2 Таня или Алена по ссылке (с user в URL)

1. Открывает ссылку вида `...?room=UUID&user=alena` (или tanya).
2. Попадает на экран приветствия («Привет, Алена»). Кнопки «Сменить пользователя» нет.
3. Нажимает «Начать аудит» → экран заполнения, часть 1. Справа — песочница (чат с Claude), если задан OpenRouter key.
4. Заполняет три части опросника. Ответы автосохраняются в Supabase (tanya_data или alena_data по роли). Видит статус партнёра: «не начала», «заполняет», «закончила».
5. «Назад» — переход на предыдущую часть (или на приветствие с части 1). «Выбрать комнату» — возврат на экран выбора комнаты (но сменить пользователя по-прежнему нельзя).
6. «Перейти к сравнению»: если партнёр не закончил — показывается модальное окно «Всё равно перейти?»; при подтверждении или если партнёр уже закончил — загружаются оба набора данных, URL переключается на страницу сравнения, отображаются баллы, таблица совпадений и вердикт (GO / PIVOT / NO-GO и т.п.).
7. На странице сравнения можно скачать отчёт (MD/JSON), скопировать ссылку на сравнение, вернуться к заполнению.

### 5.3 Админ без user в URL (выбор пользователя)

1. Открывает с `?admin=SECRET` и `?room=UUID` (или без room — тогда создаётся новая комната).
2. Экран выбора: две карточки «Таня» и «Алена», статусы по комнате, список всех комнат с данными.
3. Выбирает «Таня» или «Алена» → экран приветствия, далее как в сценарии 5.2, но с возможностью «Сменить пользователя» (возврат на экран выбора в этой же комнате).

### 5.4 Пользователь без комнаты в ссылке (не админ)

1. Открывает сайт без `room` и без админского секрета.
2. Видит экран «Нет комнаты»: сообщение «Получите ссылку у организатора» и поле для вставки ID комнаты. После ввода ID и «Войти» — переход в эту комнату (без user, дальше показывается экран выбора пользователя, но без кнопки «Создать новую»).

### 5.5 Песочница (чат с Claude)

- Доступна на экранах «Привет» и «Заполнение» в правой панели.
- История чата хранится в Supabase: `tanya_chat` / `alena_chat` по комнате и текущему пользователю. Загружается при входе в заполнение, сохраняется при отправке сообщений.
- Контекст в запросах к Claude: номер и название части опросника, при необходимости — текст текущего вопроса.

### 5.6 Логи (только админ)

- Кнопка «Логи» в левом верхнем углу. Открывает панель с последними событиями приложения (инициализация, сохранение, загрузка, ошибки). Данные только в памяти браузера, в Supabase не пишутся.

## 6. Данные опросника и сравнения

- **Структура опросника:** три части (личный аудит, бизнес-видение, сценарный анализ). В каждой — блоки и вопросы с id (q1, q2, q3_comfort, …, scenario_a, q18a, …). Типы полей: textarea, number, radio, multicheck, compound. Всё это задаётся в конфиге вопросов на фронте, в БД хранятся только пары «id вопроса → значение».
- **Подсчёт совместимости:** на экране сравнения загружаются `tanya_data` и `alena_data`, по списку параметров (все участвуют) считается совпадение: полное, частичное (в т.ч. по тексту — порог схожести), отсутствие. Критические параметры имеют больший вес. Итог — баллы и вердикт (GO / PIVOT / NO-GO и варианты).
- **Экспорт:** «Мои ответы» — MD или JSON по своим данным; «Полный отчёт» на экране сравнения — MD или JSON с обоими наборами, баллами, вердиктом и таблицей сравнения. Загрузка из файла (JSON) для восстановления своих ответов в текущей комнате.

---

## 7. Деплой и репозитории

- Исходный код: папка `founders-vision-tool` в общем workspace. Сборка: Vite, выход в `dist/`.
- Деплой: копирование в отдельную папку `founders-vision-tool-deploy`, откуда делается commit и push в GitHub (репозиторий, привязанный к Vercel). Vercel собирает проект из этого репозитория и подставляет переменные окружения при сборке. Итоговый фронт отдаётся как статика, запросы к Supabase и OpenRouter идут с клиента.

---

## 8. Кратко: что где лежит

| Что | Где |
|-----|-----|
| Ответы Тани/Алены по комнате | Supabase: `vision_rooms.tanya_data`, `vision_rooms.alena_data` |
| Чаты с Claude по комнате и пользователю | Supabase: `vision_rooms.tanya_chat`, `vision_rooms.alena_chat` |
| Текущая комната и пользователь | URL: `room`, `user` |
| Флаг админа | Браузер: `sessionStorage` |
| Логи приложения | Браузер: память (кольцевой буфер), не в БД |
| Конфиг вопросов и весов сравнения | Фронт: конфиг в коде (questions, scoring) |
| Секреты и ключи | Локально и на Vercel: переменные окружения |

Этот файл можно выгрузить и использовать отдельно для обсуждения архитектуры и улучшений (разделение слоёв, роли, хранение, сценарии).
